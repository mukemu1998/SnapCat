# SnapCat 架构规范

本文件用于约束 SnapCat 后续开发的模块边界。新增功能前先判断属于哪条主线，避免继续把业务逻辑堆回窗口代码。

## 总原则

- 窗口层只负责展示、事件转发和少量 UI 状态编排，不承载纯计算、格式化、持久化和外部服务调用细节。
- 可复用的坐标、几何、输入解析、文本格式化、历史记录构建逻辑必须优先放入 `Services` 或 `ViewModels`。
- 用户配置、API Key、快捷键、主题和历史记录等个人数据不得写入源码目录。
- 新增用户可见功能必须同步考虑托盘、快捷键、主题、配置持久化和测试包验证。
- 打包产物只放在 `artifacts/`，不提交到源码。

## 四条主线

### 截图与框选引擎

负责自由框选、等待操作框、智能预框选、颜色放大镜、坐标转换、比例锁定和未来全屏画布入口。

开发要求：

- 坐标转换、边界限制、比例计算、候选区域判断放入独立服务。
- `SelectionOverlayWindow` 和 `CaptureActionSelectionWindow` 不再新增大块算法逻辑。
- 后续全屏画布应作为独立窗口/模块接入，不在现有框选窗口上硬堆。

建议模块：

```text
Canvas/
  CanvasWindow
  CanvasToolService
  CanvasLayerService
  CanvasSelectionService
  CanvasCommandService
```

### 贴图引擎

负责固定到屏幕、贴图缩放、翻转、阵列、分组、隐藏、恢复和贴图管理。

开发要求：

- 贴图窗口只处理窗口交互，阵列、缩放、持久化和分组逻辑放入服务。
- 贴图组相关操作统一走注册服务，避免多个窗口各自维护状态。
- 新增贴图功能必须考虑重启恢复和多屏位置一致性。

### OCR 与翻译管线

负责 OCR、二维码识别、本地翻译、API 翻译、翻译浮窗和历史记录写入。

开发要求：

- OCR、翻译、二维码识别必须通过接口或服务编排，不从窗口直接调用底层实现。
- API 配置属于用户数据，默认保存到用户配置目录，不写入源码。
- 长文本、上下文翻译、多 API 配置选择应在管线层扩展，不写死在浮窗控件里。

### 设置与用户数据

负责主题、快捷键、托盘、启动项、配置目录、历史记录、临时文件清理和发布包升级兼容。

开发要求：

- 配置字段新增后必须同步保存、加载、比较、摘要显示和用户配置迁移策略。
- 快捷键解析和注册分离，新增快捷键只增加注册请求，不改 Win32 注册细节。
- 覆盖升级应保留用户配置，除非用户手动删除配置目录。

## 提交与发布约定

- README 只保留当前版本简明介绍，不写长期计划。
- 发布说明使用中文，聚焦本版用户可见变化、架构整理和已知说明。
- 每次发布前至少执行 `dotnet build`、固定测试包打包和测试包启动烟测。
- GitHub 提交说明使用中文。
